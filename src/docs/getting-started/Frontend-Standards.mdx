# **Taiv Frontend Standards**

If you're picking up your first UI ticket or just want to develop for frontend in line with our standards, start here.

<br />

### Al's Principles

The goal of the toolkit is to make building frontend feel like **building with Lego**. Most common use cases should be doable with just a couple of components and props.

A few principles to keep in mind:

- **Keep things modular.** Components in `@taiv/ui` are meant to be reusable across different apps. If a component is specific to one page or one use case, it belongs in that app's repo — not the toolkit.
- **Prefer variants over custom styles.** If something needs to look a certain way, we use 'variants' to keep things consistent everywhere. If you find yourself reaching for a CSS override, that's usually a sign the toolkit is missing something — flag it.
- **Simple but flexible.** We should have props to override, but sensible defaults and shortcuts to keep things simple.

---

### Design Constants

Before you hardcode any colors, font sizes, or spacing values, know that we have a system in place for all of these:

- **Colors** — We use a [color system](/docs/design-colors--docs) already in place. Don't hardcode hex colors or RGB values.
- **Texts** — See the [font constants](/docs/design-fonts--docs) page for more information - you should never be using raw HTML tags.
- **Spacing** — Use `rem` for values like spacing, font sizes, etc. (it scales with the user's browser settings for accessibility). Use `px` only for fixed values like borders and box shadows.

### General Workflow

This Storybook serves as a catalog of our components. When you pick up a UI ticket to build something from scratch, the general approach is:

1. **Refer to the Figma**: If there is a Figma design, make note of the exact values used for spacing, font sizes, etc.
2. **Find relevant components.** Search through Storybook before building anything - make note of what components can be reused, what can be extended, and what you need to build from scratch.
3. **Build the layout.** Use layout components to get the structure in place (more on this below).
4. **Plug in your components.** Pass in the data and any props you need.
5. **Optimize.** Handle relevant states (i.e. loading, error, empty), and add responsive behavior if the ticket calls for it.

Every component's Storybook page will provide you with the necessary import statement and usage.

### Building Layouts

`<div>` tags are ugly. You should never need them. We have three components that cover 90% of use cases:

- **Stack** — vertical (column) layout
- **Group** — horizontal (row) layout
- **Center** — centers its children

Think top-to-bottom. Start with the outermost container and break the page down into rows and columns using these. This keeps layout code readable and avoids a mess of flexbox divs and arbitrary margins.

Almost every component we have supports props such as `p` for padding, `m` for margin, `w` for width, `h` for height, etc. You can also specifically target my, mb, mx, my, etc. for more granular control of each specific direction.
If you need any specific overrides, handle it inline rather than adding a CSS class.

We also have a lot of other useful layout components - i.e. [AutoGrid](?path=/docs/components-layout-autogrid--docs), [Grid](?path=/docs/components-layout-grid--docs), [Box](?path=/docs/components-layout-box--docs), etc.

### Styling Components

Most components come with **variants** and **sizes** built in. Use them instead of writing style overrides.

- **Variants** control the visual style of a component
- **Sizes** control spacing and dimensions

Both of these work together to style a component. If a variant or size you need doesn't exist, that's a conversation about updating the toolkit more so than directly styling the component.

Most component docs will have info on how to custom style components if you need, but treat custom styling as the exception, not the norm.

---

### Handling States & UX

Looking good is not the same as working well. A huge part of good UX is just giving the user feedback about what's happening, and prevents a ton of issues down the line.

- **Loading** — If something is loading, say so. See [Loader](/docs/components-layout-loader--docs) for more information.
- **Empty states** — If there's no data to render, say so.
- **Disabled states** — Disable buttons and inputs until they're ready. A form that isn't filled yet shouldn't even be able to be submitted.
- **Notifications** — Use our [Notifications System](?path=/docs/hooks-notifications--docs) to give feedback when things go right or wrong.
- **Confirmation Modals** — Ask the user for confirmation before performing actions. Use our [Modal System](?path=/docs/hooks-confirmation-modal--docs) to do this with one line of code.
- **Tooltips** — Add [tooltips](/docs/components-info-tooltips--docs) to anything that might warrant some extra explanation.

### Documentation

If you add a new component, please add a new story to the component's Storybook page. You do **not** need to put a serious amount of effort into this.

Your doc can be a one-shot ChatGPT/Cursor/Claude prompt until we have more time to write a proper one. For now, just leave a note in that Storybook page's description that flags
the doc as AI generated (so that other devs arent' confused if there's a mistake).
